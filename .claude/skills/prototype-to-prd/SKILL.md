# Prototype → PRD 生成器

## 概述

对一个可运行的原型进行逆向工程，生成**一组结构化的 PRD 文件**供 vibe-coding 工程师参照实现。每份文件不超过 300 行，降低工程师阅读时的 token 消耗。

**核心原则：** 以原型代码为事实来源。只写代码中观察到的内容，不凭空添加。遇到任何不确定的地方，立即和用户确认，不自行猜测。

## 使用时机

- 用户有原型 / demo app，想移交给工程师实现
- 需要在正式重建之前，记录现有 UI 的功能

**不使用的场景：**
- 项目尚无任何代码（改用 brainstorming skill）
- 原型是静态图片 / Figma 文件（直接阅读即可，无代码可探索）

## 流程

### Step 1 — 定向（一条消息）

在派发 agent 之前，先与用户确认：
1. 输出语言（默认与原型注释/命名语言一致）
2. 项目根目录和关键目录（`src/`、`client/`、`app/` 等）
3. 产品名称（用于文件命名，例如 `navix`）
4. 是否有需要重点关注的功能/页面，或"全量覆盖"
5. PRD 输出目录（默认 `docs/prd/`）

有任何不确定的地方就问用户，**不要猜**。

### Step 2 — 并行探索（同时派发全部 4 个 agent）

在**同一条消息**中使用多个 Task tool 调用，**同时**派发以下四个 subagent：

- **Agent A** — 导航与页面（见下方 Prompt 模板）
- **Agent B** — 功能与用户交互
- **Agent C** — 容器数据字段（每个页面/Modal/Drawer 展示或收集的字段）
- **Agent D** — UI 组件与设计规范

### Step 2.5 — 规划文档结构（探索完成后立即执行）

四个探索 agent 全部返回后，**先规划，再写作**。

根据探索发现，判断每个内容域的体量与内聚性，制定文件清单：

**必须包含：**
- `<name>-index.prd.md`：索引，始终只有 1 份

**内容文件数量由以下规则决定：**

| 判断依据 | 拆分 | 合并 |
|---------|------|------|
| 某内容域预估 > 300 行 | 按功能模块拆分为多份 | — |
| 某内容域预估 < 50 行 | — | 并入逻辑相近的文件（如并入 index） |
| 多个模块有强内聚性 | 可按模块而非按领域组织，例如 `<name>-auth.prd.md` 覆盖认证相关的 screens + features + data | — |

**常见文件结构示例（仅供参考，实际由内容决定）：**
- 极简原型（3–5 个 screen）：index + 1–2 份内容文件
- 中等复杂度（10–20 个 screen）：index + navigation（含字段）+ features + ui，约 4 份
- 复杂 app（20+ screen，多角色）：index + navigation + 按模块的 features（3–5 份）+ 独立 fields 文件 + ui，共 7–10 份

规划完成后，向用户展示文件清单（格式：文件名 | 预估内容 | 预估行数），**等用户确认后**再进入 Step 3。如有任何不确定，先问用户。

### Step 3 — 写 PRD 文件（根据规划并行派发写作 agent）

用户确认文件清单后，在**同一条消息**中并行派发对应数量的写作 subagent，每个负责写一份 `.prd.md` 文件。

每个写作 agent 拿到的信息：相关探索数据 + 文件内容职责 + 模板 + 硬性约束（≤300 行）。

### Step 4 — 校验（派发验证 agent）

全部写作 agent 完成后，派发**一个**验证 subagent（见下方 Prompt 模板），检查：
1. 每份文件行数 ≤ 300
2. Index 中的链接指向真实存在的文件
3. 无空章节（仅有标题无内容）
4. 无未填写的占位符（`[待填写]`、`TODO` 等）
5. Features 中引用的 Screen 名称在 navigation.prd.md 中均有对应定义

验证 agent 返回问题列表（文件名 + 行号 + 描述）或 `PASS`。

若有问题，逐一修复后告知用户。

### Step 5 — 汇报

告知用户：
- 生成的所有文件路径
- 各文件行数
- 校验结果
- 是否有需要用户决策的待解问题

## 输出文件规范

### 文件命名规则

- 所有文件必须以 `.prd.md` 结尾
- 文件名格式：`<product-name>-<topic>.prd.md`
- `<name>-index.prd.md` 是唯一固定的文件，其余文件按 Step 2.5 规划结果确定
- 按模块拆分时，用连字符标注模块名：`<name>-features-auth.prd.md`

**文件命名词汇参考（按需组合，不强制使用）：**

| 内容 | 建议命名词汇 |
|------|------------|
| 导航、页面、路由 | `navigation` / `screens` |
| 功能规格 | `features` / `features-<模块名>` |
| 容器字段（独立文件时） | `fields` / `fields-<模块名>` |
| UI 组件、设计系统 | `ui` / `design-system` |
| 特定功能模块 | `auth` / `chat` / `profile` / `onboarding` 等 |

### index.prd.md 模板

```markdown
# [Product Name] — PRD 索引

> **Status:** Draft | **Date:** YYYY-MM-DD | **Source:** Prototype 逆向工程

## 产品概述

[2–4 句：产品是什么、面向谁、解决什么问题。]

## User Personas

| Persona | 描述 | 主要使用的 Screen |
|---------|------|-----------------|
| [名称]  | [是谁、有什么需求] | [Screen 列表] |

## 文档目录

| 文件 | 内容 | 行数 |
|------|------|------|
| [navigation.prd.md](./[name]-navigation.prd.md) | Screen 地图、导航流程、各容器数据字段 | ~N |
| [features.prd.md](./[name]-features.prd.md) | 功能规格、验收标准 | ~N |
| [ui.prd.md](./[name]-ui.prd.md) | 组件清单、Design System | ~N |

## 核心用户流程（速览）

[列出 3–5 个最重要的流程，每条一行，格式：流程名 → Screen A → Screen B → Screen C]

## 待解问题

- [ ] [逆向工程中发现的模糊点或需要产品确认的内容]
```

### navigation.prd.md 模板

```markdown
# [Product Name] — 导航与页面

> 索引：[index.prd.md](./[name]-index.prd.md)

## Screen 地图

[用树形结构列出所有 screen，标注路径或入口方式]

## 导航流程

### Flow: [名称]
1. 用户进入 [Screen A]
2. 点击 [元素] → 进入 [Screen B]
3. ...终态：[用户完成了什么]

## 容器规格

对每个页面、Modal、Drawer、Sheet 单独列出，格式如下：

---

### [容器名称]（页面 / Modal / Drawer / Sheet）

**用途：** [一句话说明这个容器的作用]
**触发方式：** [如何进入 / 打开]（页面填写"路由直接访问"或上游来源）
**出口：** [用户可以从这里去哪里，或关闭后回到何处]

**界面线框：**

```
+----------------------------+
| < 返回         标题         |
+----------------------------+
|  [头像]   用户名             |
|           简介文本...        |
+----------------------------+
|  标签1  标签2  标签3         |
+----------------------------+
|  列表项 A               >  |
|  列表项 B               >  |
|  列表项 C               >  |
+----------------------------+
|     [ 主操作按钮 ]          |
+----------------------------+
```

> 线框只需体现布局骨架，不需要精确。使用 ASCII 字符绘制（见下方规范）。

**数据字段：**

| 字段标签 | 类型 | 必填 | 可编辑 | 说明与约束 |
|---------|------|------|--------|-----------|
| [用户看到的标签名] | 文本 / 数字 / 日期 / 图片 / 开关 / 单选 / 多选 / 富文本 | 是/否 | 是/否 | [展示说明、取值范围、格式要求等，用自然语言] |

> 无数据字段的纯展示/导航容器可省略字段表。
```

### features.prd.md 模板

```markdown
# [Product Name] — 功能规格

> 索引：[index.prd.md](./[name]-index.prd.md)

## F-[N]: [功能名称]

**Screen：** [所在 screen]
**User story：** 作为 [persona]，我想要 [操作]，以便 [结果]。

**验收标准：**
- [ ] [具体可测试的条件]

**交互细节：**
- [触发方式]：[发生了什么]
- [边缘情况]：[发生了什么]

**状态：**
- 默认 / Loading / 空状态 / 错误：[各自的 UI 描述]
```

### fields.prd.md 模板（容器字段独立文件，仅当字段体量大时使用）

当字段数量过多导致 navigation.prd.md 超出 300 行时，可将字段单独拆出：

```markdown
# [Product Name] — 容器数据字段

> 索引：[index.prd.md](./[name]-index.prd.md)

## [容器名称]

**所属页面：** [所在页面名称]
**容器类型：** 页面 / Modal / Drawer / Sheet

| 字段标签 | 类型 | 必填 | 可编辑 | 说明与约束 |
|---------|------|------|--------|-----------|
| [标签] | 文本 / 数字 / 日期 / 图片 / 开关 / 单选 / 多选 / 富文本 | 是/否 | 是/否 | [自然语言描述] |

---

## [下一个容器]
...
```

> **类型词汇说明：** 使用产品语言，不使用代码类型名。
> - 文本（短文本、长文本）、数字（整数、小数）、日期、时间、日期时间
> - 图片、文件、链接、颜色
> - 开关（布尔值）、单选、多选、评分
> - 富文本、Markdown

## ASCII 线框规范

线框使用纯 ASCII 字符，在任何终端和 Markdown 渲染器中均可正确显示。

### 框架字符

```
+-------+   ←  边框（+ 转角，- 横线，| 竖线）
|       |
+-------+
```

### 元素速查表

| 元素 | 写法 | 示例 |
|------|------|------|
| 普通按钮 | `[ 文字 ]` | `[ 确认 ]` `[ 取消 ]` |
| 主操作按钮（全宽） | `[===== 文字 =====]` | `[===== 发布 =====]` |
| 输入框（空） | `[________________]` | 单行文本输入 |
| 输入框（有值） | `[已填写的内容____]` | |
| 图片 / 头像占位 | `[IMG]` / `[AVT]` | |
| 开关（关） | `[ ] 标签` | |
| 开关（开） | `[x] 标签` | |
| 单选 | `(o) 选中` `( ) 未选` | |
| 下拉选择 | `[选项文字        v]` | |
| 列表项带箭头 | `列表项文字      >` | |
| 加载中 | `[... 加载中 ...]` | |
| 空状态 | `( 暂无内容 )` | |
| 文字内容省略 | `文字文字...` | |
| 底部标签栏 | `[首页] [发现] [我的]` | |
| 分割线 | `- - - - - - - - -` | |

### 宽度参考

- 手机屏幕：30 字符宽（含两侧边框）
- 平板 / Web 侧边栏：24 字符宽
- Web 全宽卡片：50 字符宽
- Modal / Sheet：26 字符宽

### 典型示例

**手机页面：**
```
+------------------------------+
| < 返回          页面标题      |
+------------------------------+
|  [AVT]   用户昵称             |
|          签名文字...          |
+------------------------------+
|  [x] 接收通知                |
|  [ ] 公开主页                |
+------------------------------+
|  地区             [选择    v] |
|  生日             2000-01-01 |
+------------------------------+
|  [========= 保存 =========]  |
+------------------------------+
```

**底部 Sheet：**
```
      +------------------------+
      |   - - - (拖动条) - - - |
      +------------------------+
      |  标题文字               |
      |  说明文字，可多行...     |
      +------------------------+
      |  选项 A             >  |
      |  选项 B             >  |
      |  选项 C             >  |
      +------------------------+
      |      [ 取 消 ]         |
      +------------------------+
```

**居中 Modal：**
```
  +------------------------+
  |  标题                  |
  |                        |
  |  说明文字，解释操作的   |
  |  后果或所需信息。       |
  |                        |
  |  [ 取消 ]  [=确认=]    |
  +------------------------+
```

## Agent Prompt 模板

### Agent A — 导航与布局探索

```
在 [PROJECT ROOT] 探索原型 codebase，梳理每一个 screen/page/route 的导航关系和视觉布局结构。

只读代码库内容，不参考外部文档。

1. 找到所有路由定义（Router、Route、createBrowserRouter、useRouter、Stack.Screen、app/ 或 pages/ 文件路由）
2. 对每个 screen 记录：文件路径、推断用途、与哪些 screen 有导航关系
3. 列出所有 modal/drawer/sheet/overlay 及其触发位置
4. 识别角色区分的 screen（admin vs user vs guest）
5. 对每个容器（页面/Modal/Drawer），描述其视觉布局结构：
   - 纵向区域划分（顶部 header、滚动主体、底部 bar 等）
   - 每个区域内有哪些主要 UI 块（卡片、列表、表单、图片、按钮等）
   - 横向布局（单列、两列、网格等）
   - 整体是否可滚动，还是固定高度

关注领域：[FOCUS AREAS 或 "全量覆盖"]

返回：screen 扁平列表（含路径、用途、导航关系）+ 每个容器的布局结构描述 + 弹层列表。
```

### Agent B — 功能探索

```
在 [PROJECT ROOT] 探索原型 codebase，整理每一个用户可见的功能和交互。

只读代码库内容，不参考外部文档。

1. 找到所有交互元素：onClick、form 提交、gesture handler
2. 对每个交互记录：触发条件、结果（state 变化/导航/API 调用）、成功/错误状态
3. 提取所有表单字段：name、type、校验规则（zod、yup、内置 HTML）
4. 找到条件渲染展示的不同 UI 状态（loading/empty/error/success）
5. 识别多步骤流程（wizard、onboarding、confirmation dialog）

关注领域：[FOCUS AREAS 或 "全量覆盖"]

返回：按 screen 分组的功能列表（含交互细节、边缘情况、UI 状态）。
```

### Agent C — 容器字段探索

```
在 [PROJECT ROOT] 探索原型 codebase，为每个页面、Modal、Drawer、Sheet 梳理其展示或收集的数据字段。

只读代码库内容，不参考外部文档。输出中不得包含任何代码（不写 TypeScript、JSON、API 路径）。

对每个容器（页面/Modal/Drawer/Sheet）完成以下工作：
1. 找到该容器渲染的所有数据字段——无论是展示字段还是表单输入字段
2. 对每个字段记录：
   - 字段标签：用户界面上显示的文字标签（从 label、placeholder、aria-label 或临近文字推断）
   - 数据类型：用产品语言描述（文本/数字/日期/图片/开关/单选/多选/富文本等），不写代码类型
   - 是否必填：从 required 属性、zod schema 或校验逻辑推断，用"是/否"表达
   - 是否可编辑：区分只读展示字段和可编辑输入字段
   - 说明与约束：字符限制、取值范围、格式要求、特殊业务规则——用自然语言描述
3. 忽略纯装饰性、纯布局性内容（图标、分割线、背景图等）
4. 对字段来源来自 mock/fixture 数据的，记录该数据所在文件路径（仅作内部参考，不写入 PRD）

关注领域：[FOCUS AREAS 或 "全量覆盖"]

返回：按容器分组的字段清单（容器名称 → 字段列表）。
```

### Agent D — UI 探索

```
在 [PROJECT ROOT] 探索原型 codebase，盘点 UI 组件体系。

只读代码库内容，不参考外部文档。

1. 列出 components/ 或 ui/ 目录中的所有共享组件：name、key props、variants
2. 识别 design system（shadcn/ui、MUI、Ant Design、Radix、Tailwind-only、custom）
3. 找出重复出现的布局模式（navigation bar、header、tab bar、sidebar）
4. 提取 design token：CSS custom properties、Tailwind config、theme 对象——颜色、typography、spacing
5. 记录无障碍处理：aria-label、role、tabIndex、focus management

返回：组件清单表格 + design system 概要。
```

### 写作 Agent（W-Index / W-Nav / W-Features / W-Data / W-UI）

```
根据以下探索数据，按照指定模板写 PRD 文件，写入 [OUTPUT PATH]。

探索数据：
[粘贴对应 agent 的原始输出]

模板要求：
[粘贴对应模板内容]

硬性约束：
- 文件行数严格不超过 300 行
- 超出时按模块拆分为多个文件，并在 index 中更新链接
- 只写代码中观察到的内容，不添加推测内容
- PRD 中禁止出现任何代码内容：不写 TypeScript/JavaScript 代码块、不写 JSON 示例、不写 API 路径（如 GET /api/users）、不写变量名或函数名
- 数据字段的类型用产品语言描述（文本、数字、日期、图片……），不用代码类型（string、number、Date……）
- navigation 类文件的每个容器规格中，必须包含 ASCII 线框图（使用本 skill 的 ASCII 线框规范），线框放在字段表之前
- 所有文件以 .prd.md 结尾
- PRD 中严禁出现任何代码内容：不写 TypeScript / JavaScript 代码块、不写 JSON 示例、不写 API 路径、不写代码类型名（用"文本""数字"等产品语言替代 string/number）

完成后返回：文件路径 + 实际行数。
```

### 验证 Agent

```
验证 [OUTPUT DIR] 中的所有 .prd.md 文件。

检查项：
1. 行数：每个文件 ≤ 300 行（超出则报告文件名 + 实际行数）
2. 索引完整性：index.prd.md 中的每个链接对应的文件都存在
3. 空章节：找出只有标题没有实际内容的章节
4. 占位符：找出未填写的 [待填写]、TODO、[名称]、[描述] 等
5. 交叉引用：features 类文件中引用的 Screen 名称在 navigation/screens 类文件中均有定义
6. 代码内容：找出不应出现在 PRD 中的代码内容——TypeScript/JavaScript 代码块（interface、type、const、function 等关键词）、JSON 代码块、API 路径格式（GET /api/...）、变量名或函数名（camelCase 或 snake_case 的技术命名）

返回格式：
- 若全部通过：PASS
- 若有问题：按文件列出（文件名 | 行号 | 问题描述）
```

## 常见错误

**PRD 中写入代码：** PRD 面向产品和工程师沟通需求，不是技术规格文档。禁止出现 TypeScript 代码块、JSON 示例、API 路径、变量名等任何代码内容。数据字段的类型用产品语言（文本、数字、日期……），校验规则用自然语言（"最多20字"、"仅限正整数"），不用代码描述。

**跳过规划直接写作：** Step 2.5 是关键决策点，不能省略。文件数量和结构必须在写作前确定并经用户确认，写完再重组代价极高。

**按领域而非按内聚性拆分：** 若认证功能的 screens、features、data 体量都很小，应合并为一个 `<name>-auth.prd.md`，而非强行拆成三份。以读者的阅读效率为准。

**凭空发明需求：** 只写代码中可见的内容。占位符内容（`Coming soon`、硬编码假数据）记入 index 的"待解问题"或对应文件的"范围外"章节。

**忽略空状态：** 空状态 UI（无数据、首次使用）往往是最重要的边缘情况，Agent B 必须覆盖。

**预估不准导致超限：** 探索 agent 的输出量通常比最终文档大 2–3 倍，规划时应保守估计行数。若写作 agent 返回行数超限，立即重新拆分，不要强塞。

**验证跳过：** 不要省略 Step 4 的验证 agent，它是保证文档可用性的最后防线。
